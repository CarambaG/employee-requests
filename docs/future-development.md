# Возможное развитие системы

## Согласование заявки руководителем

В бизнес-процесс следует добавить этап согласования, например:

```text
Новая -> На согласовании -> В работе -> Выполнена
                |-> Отклонена
```

Понадобятся новые статусы, связь сотрудника с руководителем и операция принятия решения. Решение руководителя лучше хранить отдельной сущностью `request_approvals` с заявкой, согласующим, результатом, комментарием и временем решения. Проверку переходов следует расширить так, чтобы перевод в работу был возможен только после успешного согласования.

## Несколько исполнителей

Поле `assignee_id` в `requests` перестанет соответствовать отношению один-ко-многим. Его следует заменить таблицей связи:

```text
request_assignees
- request_number
- employee_id
- assigned_at
- assigned_by
```

Первичный ключ может состоять из `request_number` и `employee_id`. Фильтрация и отчётность должны учитывать эту таблицу. Также потребуется определить правило завершения: заявку закрывает любой исполнитель, ответственный исполнитель или только после завершения работы всеми участниками.

## История изменения статусов

Следует добавить таблицу `request_status_history`:

```text
- id
- request_number
- from_status_id
- to_status_id
- changed_by
- changed_at
- comment
```

Запись текущего статуса и добавление строки истории должны выполняться в одной транзакции. Текущий `status_id` в `requests` можно оставить для быстрого чтения, а историю использовать для аудита и расчёта времени нахождения на каждом этапе.

## Сроки в зависимости от типа заявки

Потребуется справочник `request_types` с названием, сроком выполнения и при необходимости правилами приоритета. В `requests` добавляется `type_id`, а `due_at` вычисляется при создании на основании типа и даты создания. Для изменения нормативов без влияния на уже созданные заявки рассчитанный срок следует сохранять непосредственно в заявке.

## Разграничение прав доступа

Необходимы аутентификация и роли:

- сотрудник создаёт заявки и просматривает доступные ему заявки;
- руководитель согласует заявки своего подразделения и назначает исполнителей;
- администратор управляет справочниками, пользователями и ролями.

Для хранения прав можно использовать таблицы `users`, `roles` и `user_roles`. В HTTP-слой добавляется middleware аутентификации, а проверки доступа выполняются в сервисном слое, чтобы правила нельзя было обойти через другой транспорт. Для чувствительных операций полезно вести журнал аудита.
