# Проверка производительности

## Целевой запрос

Необходимо получить все просроченные заявки конкретного исполнителя, находящиеся в статусе `in_progress`, и отсортировать их по сроку выполнения:

```sql
SELECT
    r.number,
    r.created_at,
    r.author_id,
    r.assignee_id,
    r.description,
    r.due_at,
    r.status_id,
    r.updated_at
FROM requests AS r
WHERE r.assignee_id = $1
  AND r.status_id = (
      SELECT id
      FROM request_statuses
      WHERE code = 'in_progress'
  )
  AND r.due_at < NOW()
ORDER BY r.due_at;
```

## Исходное состояние

В исходной схеме таблица `requests` имеет только индекс первичного ключа по `number`. Специального индекса для фильтрации по исполнителю, статусу и сроку выполнения пока нет. Это намеренно: сначала необходимо зафиксировать план и время выполнения до оптимизации.

## Подготовка данных

```bash
make up
make seed
make seed-verify
```

Полный генератор создаёт 1000 сотрудников и 1 000 000 заявок.

## Замер до оптимизации

```bash
make benchmark-baseline
```

Команда:

1. проверяет наличие данных;
2. выбирает конкретного исполнителя с наибольшим числом просроченных заявок в работе, если `ASSIGNEE_ID` не задан;
3. выполняет один прогревочный запуск;
4. выполняет три измеряемых `EXPLAIN (ANALYZE, BUFFERS)`;
5. сохраняет отсортированный результат запроса в CSV.

Файлы результата:

```text
performance-results/baseline.txt
performance-results/baseline_rows.csv
```

Для фиксированного исполнителя:

```bash
ASSIGNEE_ID=42 make benchmark-baseline
```

Для сравнения следует использовать одинаковый `ASSIGNEE_ID`, одинаковый набор данных и три измеряемых запуска после прогрева. В отчёте важны:

- тип доступа к `requests` (`Seq Scan`, `Index Scan` или `Bitmap Scan`);
- фактическое число обработанных и отброшенных строк;
- операция сортировки и используемая память;
- чтения из буферов;
- `Execution Time`.

Результаты в репозитории не зафиксированы заранее, поскольку время зависит от процессора, диска, настроек Docker и объёма памяти конкретной машины. Команда формирует воспроизводимый отчёт непосредственно в окружении, где проводится проверка.
