# Проверка производительности

## Целевой запрос

Необходимо получить все просроченные заявки конкретного исполнителя, находящиеся в статусе `in_progress`, и отсортировать их по сроку выполнения:

```sql
SELECT
    r.number,
    r.created_at,
    r.author_id,
    r.assignee_id,
    r.description,
    r.due_at,
    r.status_id,
    r.updated_at
FROM requests AS r
WHERE r.assignee_id = $1
  AND r.status_id = (
      SELECT id
      FROM request_statuses
      WHERE code = 'in_progress'
  )
  AND r.due_at < NOW()
ORDER BY r.due_at;
```

## Подготовка данных

```bash
make up
make seed
make seed-verify
```

Полный генератор создаёт 1000 сотрудников и 1 000 000 заявок.

## Замер до оптимизации

На коммите до добавления индекса выполните:

```bash
make benchmark-baseline
```

Сценарий:

1. проверяет наличие данных;
2. выбирает исполнителя с наибольшим числом подходящих заявок, если `ASSIGNEE_ID` не задан;
3. выполняет один прогревочный запуск;
4. выполняет три измеряемых `EXPLAIN (ANALYZE, BUFFERS)`;
5. сохраняет отсортированный результат запроса в CSV.

Файлы результата:

```text
performance-results/baseline.txt
performance-results/baseline_rows.csv
```

Для фиксированного исполнителя:

```bash
ASSIGNEE_ID=42 make benchmark-baseline
```

В исходной схеме у `requests` отсутствует индекс, соответствующий фильтрам целевого запроса. На миллионе строк планировщик обычно вынужден просматривать значительную часть таблицы, отбрасывать неподходящие записи и отдельно сортировать результат. Конкретный план необходимо брать из сформированного `baseline.txt`.

## Выполненная оптимизация

Добавлена миграция `000002_optimize_overdue_assignee_query`:

```sql
CREATE INDEX requests_assignee_status_due_at_idx
    ON requests (assignee_id, status_id, due_at);
```

Поля расположены в следующем порядке:

1. `assignee_id` — равенство по конкретному исполнителю и наиболее селективное условие;
2. `status_id` — равенство по статусу `in_progress`;
3. `due_at` — диапазон `due_at < NOW()` и одновременно порядок результата.

Индекс ускоряет запрос по двум причинам:

- PostgreSQL читает только диапазон записей нужного исполнителя и статуса вместо полного просмотра миллиона строк;
- записи в индексе уже упорядочены по `due_at`, поэтому отдельная операция `Sort` обычно не требуется.

В индекс намеренно не включено поле `description`: оно может быть большим, заметно увеличило бы размер индекса и стоимость вставок и обновлений. После поиска подходящих ссылок на строки PostgreSQL получает полные данные из таблицы.

## Замер после оптимизации

На этом коммите выполните:

```bash
make benchmark-optimized
```

Цель автоматически выполняет миграции, проверяет наличие индекса и использует `Assignee ID` из `performance-results/baseline.txt`. Поэтому baseline-файл следует сохранить после первого теста.

При необходимости исполнитель задаётся явно:

```bash
ASSIGNEE_ID=42 make benchmark-optimized
```

Файлы результата:

```text
performance-results/optimized.txt
performance-results/optimized_rows.csv
performance-results/comparison.md
```

CSV-файлы создаются для проверки сортировки и состава выборки, но исключены из Git как вспомогательные результаты. Для отчёта в репозитории достаточно сохранить `baseline.txt`, `optimized.txt` и `comparison.md`.

`comparison.md` автоматически рассчитывает:

- среднее время трёх измеряемых запусков до оптимизации;
- среднее время трёх запусков после оптимизации;
- коэффициент ускорения;
- снижение времени выполнения в процентах;
- основные операции планов выполнения.

## Условия корректного сравнения

Для сравнения необходимо использовать:

- одну и ту же базу и один набор из 1 000 000 заявок;
- одинаковый `ASSIGNEE_ID`;
- одинаковые ресурсы Docker;
- три запуска после одного прогрева;
- отсутствие перезаполнения базы между тестами.

Запрос использует `NOW()`, поэтому при большом интервале между замерами число просроченных строк может немного измениться. Для тестовых данных со сроками, распределёнными по дням, это обычно не влияет на вывод о плане, но количество найденных строк следует указать в итоговом описании.

Фактические значения времени зависят от процессора, диска, памяти, версии PostgreSQL и настроек Docker, поэтому они формируются в окружении проверяющего, а не задаются заранее.
