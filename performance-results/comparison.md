# Сравнение производительности

- PostgreSQL: `17.10`
- Сотрудников: `1000`
- Заявок: `1 000 000`
- Исполнитель: `21`
- Подходящих заявок: `962`
- Среднее время до оптимизации: `27.764 ms`
- Среднее время после оптимизации: `2.354 ms`
- Ускорение: `11.79x`
- Снижение времени выполнения: `91.52%`
- Размер индекса: `39 MB`

## План до оптимизации

```text
Gather Merge
  -> Sort by r.due_at
       -> Parallel Seq Scan on requests
```

Таблица `requests` просматривалась параллельно почти полностью. В каждом worker было отброшено около 333 тысяч неподходящих строк, после чего результат сортировался по сроку выполнения.

## План после оптимизации

```text
Sort by r.due_at
  -> Bitmap Heap Scan on requests
       -> Bitmap Index Scan on requests_assignee_status_due_at_idx
```

Составной индекс `(assignee_id, status_id, due_at)` позволил получить ссылки только на 962 подходящие строки вместо полного просмотра миллиона заявок. PostgreSQL выбрал bitmap heap scan, поэтому отдельная сортировка сохранилась: bitmap-доступ не гарантирует порядок B-tree. Сортировка выполнялась уже над небольшой выборкой и заняла незначительную часть общего времени.

Основной эффект оптимизации — сокращение объёма чтения и числа проверяемых строк, а не устранение операции `Sort`.
